# 案例复盘 · 网易云音乐控制插件从 0 到 1

> 完整还原本次开发的时间线、每个岔路口的决策与理由。这是“流程”本身，比结论更值得学。

---

## 阶段一：报障与诊断（别急着动手修）

**用户报障**：研究给网易云装插件启动 MCP 服务来让 AI 控制它，但“装了插件后网易云无法启动”，
要求解决。

**我没有立刻去“修”，而是先收集事实（只读，不改动）**：

1. 定位安装目录与版本：注册表 → `D:\CloudMusic\CloudMusic`，版本 **3.1.32.205206**。
2. 检查程序目录被改动的痕迹：列目录按修改时间排序 → 核心 `cloudmusic.exe/.dll` 仍是安装
   时间戳，**没有** BetterNCM 常见劫持 DLL（`msimg32.dll`/`dwmapi.dll`），没有 `betterncm` 目录。
3. 检查隐蔽注入位置：IFEO 调试器劫持、`AppInit_DLLs`、卡死进程、Windows 事件日志 → **全干净**。

**关键澄清**：用户补充“现在是移除插件状态，是可以启动的”。

> **诊断定性**：程序目录和注册表都干净 ⇒ 插件没在目录留下东西 ⇒ 问题不是“残留要清理”，
> 而是**插件本身与该版本不兼容**：装上就崩，移除就好。

📌 **教训**：报障的字面（“无法启动”）和真实根因（“某版本下注入被反篡改拦截”）往往不同。
**先复现+收集证据，再定性，最后才谈方案。** 切忌一上来就“清理残留”。

---

## 阶段二：把根因查透（用证据而非记忆）

联网核实（因为该版本比知识库新）：
- 旧 **BetterNCM 1.3.x** 用 **DLL 劫持**注入，只适配到网易云 **3.0.x**。
- 网易云 **3.1.x** 加了**启动完整性校验**，发现目录被改/进程有外来 DLL 就拒绝启动。
- **Chromatic**（BetterNCM 重写继任者）尚不完善，自动安装器明确报“此版本不支持”。

> **根因**：失败的不是“注入器写得差”，而是**注入手段（改文件/塞 DLL）正面撞上反篡改**。

---

## 阶段三：路线决策（这一步决定成败）

我给出两条路并讲清权衡：
- **A. 无侵入外部控制**（媒体键 / SMTC 系统媒体会话 / `orpheus://` 协议）——稳、抗更新，但深度有限。
- **B. 侵入式**（BetterNCM/Chromatic 或自写注入）——深，但脆、撞反篡改。

我先核实了 `orpheus://` 协议已注册（无侵入路可行），用户**先选了 A（无侵入 MCP）**。

刚要落地 A，用户**改主意要侵入式**，并问“Chromatic 能用吗 / 能自写注入器吗”。

> **关键认知跃迁**：用户要的“侵入式”本质是**深度控制 App 内部**，不等于“必须写 DLL”。
> 我意识到——**换一个不改文件的“注入层”就能既深又稳**：
> 网易云是 CEF 应用，用 Chromium 自带的 **远程调试协议 (CDP)** 就能在它进程内执行任意 JS，
> 拿到和 BetterNCM 同级的能力，却**不碰任何文件**，因此根本不触发那道启动保护。

📌 **教训**：不要被用户（或自己）的“实现词”框住（“注入器”“DLL”）。回到**真实意图**
（“深度控制内部”），再在**意图**层面找更优解。

---

## 阶段四：决定性验证（一票否决式的关键实验）

整条路成立与否，全看一件事：**网易云这个 CEF 编译版是否允许 `--remote-debugging-port`**
（有些加固版会裁掉）。于是先做这个**最高价值实验**：

```powershell
& "D:\CloudMusic\CloudMusic\cloudmusic.exe" --remote-debugging-port=9222
Invoke-RestMethod http://127.0.0.1:9222/json/version
```

结果：端口可用，返回 `Chrome/91`、`NeteaseMusicDesktop/3.1.32.205206`，并列出页面目标。
✅ 路通了——走轻松的 CDP；否则才需要考虑“启动后注入”等硬核手段。

📌 **教训**：识别出**能一票否决整个方案的假设**，**最先**验证它。别在不确定地基上盖楼。

---

## 阶段五：逐步逆向（每一步都实测，不靠猜）

连上 page 的 websocket，用 `Runtime.evaluate` 在页面里跑 JS，层层深入：

1. **枚举全局** → 发现 `window.channel`（原生桥）、`window.webpackJsonp`（老式 webpack）、
   `APP_CONF`；`audio` 元素为 0 ⇒ 播放走原生，控制得走更高层。
2. **抠出 webpack require**（jsonp execute-push 技巧）→ `window.__wpr`，1700+ 模块在手。
3. **扫描模块缓存** → 定位音频桥(1131)、API 相关(15/12/2181)、枚举常量(4)、`PLAYMODE_MAP`(2875)；
   发现是 **dva(Redux)** 架构（模块有 `namespace/state/reducers/effects`）。
4. **找到 Redux store**（React fiber 回溯 `memoizedProps.store`）→ 读到 `playing`(当前歌)、
   `playingList`(队列)、`fmPlaying` 等切片；用户已登录，队列 32 首。
5. **提取真实 action**（正则扫模块源码里的 `"playing/xxx"` 字符串，**不是猜**）→ 拿到
   `playing/pause|resume|setVolume|...`、`playingList/jump2Track` 等全表；
   逐个读 effect 源码确认 **payload 形状**。
6. **实测控制**：派发 `playOneTrackInPlayingList`、`jump2Track {flag:1}` → 歌名真的切换。
   （中途发现 `switchResumeOrPause` 不传 `{triggerScene}` 会在 saga 内抛错——这正是第一次
   测试“无效”的原因。）
7. **定位万能 API 工厂**：所有 API 都是 `Object(r.a)({url})(params)`，扫出 **模块 12 的 `.a`**；
   用一次真实 `/api/cloudsearch/pc` 搜索验证通过。
8. **实测任意点歌**：搜索→`/api/v3/song/detail`→构建队列项→派发，成功播放队列外的歌。

📌 **教训**：逆向要**验证驱动**——每定位一个接口就立刻在运行实例上实测；**action/payload
都从源码提取而非猜测**；遇到“没反应”先怀疑**参数缺失**和**异步时序**（dva saga 是异步的，
派发后要等 200~600ms 再读 state）。

## 阶段六：落地工程化

1. **写 MCP server**（`netease-music-mcp-server.mjs`，Node 零额外依赖）：
   - `BOOTSTRAP`：每次命令前注入，幂等地建立 `__wpr/__ncm_store/__ncm_request` 三把钥匙
     （按**特征**而非写死 id 定位，抗版本变化）。
   - 23 个工具：读状态、播放控制、点歌、搜索、歌单、歌词、FM，加 `ncm_api/ncm_dispatch/
     ncm_eval` 三个万能逃生口。
2. **接入现有插件管理器**：发现项目里**已有**一个走 BetterNCM 的网易云插件（就是它在 3.1+
   抛“暂不支持”），把 `software-plugin-manager.ts` **整体改造为 CDP**（安装=复制 server+写
   MCP 配置，不下载、不注入），并同步重写测试。
3. **验证**：单测 119/119 通过、`tsgo` 类型检查通过、端到端 MCP 实测各工具生效、复制到资产
   目录后的副本也能解析依赖并控制。

📌 **教训**：落地前先摸清**项目既有架构**（MCP 配置格式、插件管理器、测试约定），**复用**
而非另起一套；改完**跑测试+类型检查+端到端**三重验证。

---

## 一图流总结

```
报障 → 收证据(只读) → 定性(版本不兼容,非残留)
     → 查透根因(注入撞反篡改)
     → 路线决策(回到"深度控制内部"的真实意图)
     → 关键认知:换"不改文件的注入层"= CDP
     → 一票否决实验:CDP 端口可用?  ✅
     → 验证驱动逆向: globals→require→store→actions→api工厂  (每步实测)
     → 工程化: MCP server + 接入既有插件管理器 + 三重验证
```

> 把这套流程抽象成可复用方法论，见 [02-通用方法论.md](./02-通用方法论.md)。
