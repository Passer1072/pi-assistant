# 通用方法论 · 给任意桌面应用开发 AI 控制插件

> 从本次网易云案例抽象出的、可复用于**任何**桌面应用的开发流程。配合
> [03-逆向技术手册.md](./03-逆向技术手册.md)（具体技术）和 [toolkit/](./toolkit/)（脚本）使用。

---

## 第 0 步：先判断应用类型

不同类型，控制手段完全不同。先用任务管理器/进程信息/安装目录判断：

| 线索 | 应用类型 | 主要控制手段 |
|---|---|---|
| 目录有 `libcef.dll` / `*.pak` / `cef` | **CEF** 应用（如网易云） | **CDP 远程调试**（本案例）|
| 目录有 `resources/app.asar` / `electron` | **Electron** 应用 | CDP 远程调试 / 主进程 IPC |
| 有 `.exe` + Qt/原生 DLL，无 Chromium | **原生**应用 | UI Automation / 媒体键 / 窗口消息 / 官方 API |
| 就是个网页/PWA | **Web** | 浏览器扩展 / CDP / 官方 Web API |

> CEF/Electron 是重点：它们内部跑的是网页，**能用 CDP 在进程内执行 JS**，这是“深度控制
> 但不改文件”的黄金通道。

---

## 第 1 步：控制层决策树（核心）

**按顺序选第一个能用的层。越靠前越稳、越省事；注入永远是最后手段。**

```
1. 有没有官方/专用 API 或 MCP？           → 有就用（最稳）
2. 有没有官方协议 / 调试通道？
   - CEF/Electron: --remote-debugging-port (CDP)   ← 本案例的解
   - 自定义 URL scheme (如 orpheus://)
   - 系统级: SMTC 媒体会话、媒体键、D-Bus/AppleScript
3. 外部自动化：UI Automation / 窗口消息 / 模拟输入   → 不碰内部，但较浅、易受 UI 改版影响
4. 代码注入：DLL 注入 / 改前端包 / BetterNCM 类宿主   → 最深，但最脆、易撞反篡改/封号
```

**黄金法则：能不改它的文件，就绝不改。**
- 改文件 = 触发完整性校验 / 杀软 / 自动更新覆盖 / 维护噩梦。
- “附加调试器 + 跑 JS”不改任何文件，随时可逆，抗更新。

> 本案例最初卡在第 4 层（BetterNCM 注入撞 3.1 反篡改）；退回第 2 层（CDP）后豁然开朗，
> 拿到和注入同级的能力却没有它的代价。

---

## 第 2 步：一票否决实验（最先做）

识别出**“不成立则整个方案作废”的那个假设**，最先验证它。别在不确定的地基上盖楼。

- CEF/Electron：`app.exe --remote-debugging-port=9222` 后 `curl http://127.0.0.1:9222/json/version`
  能否返回？（加固版可能裁掉调试端口。）
- 自定义协议：协议是否在注册表注册、`%1` 怎么传？
- 外部自动化：目标窗口/控件能否被 UIA 读到？

> 本案例第一刀就砍这里：CDP 端口可用 → 全盘可行。

---

## 第 3 步：验证驱动的逆向（拿到“三把钥匙”）

对 CEF/Electron 应用，目标是在页面里拿到三类句柄（具体代码见技术手册）：

1. **模块系统入口**（webpack require）——访问应用全部内部模块。
2. **状态容器**（Redux/dva/Vuex store / 或框架状态）——读状态 + 派发动作来控制。
3. **网络层**（应用自己的签名 API 调用器）——调它的后端 API（搜索/详情/列表…）。

**铁律**：
- **每定位一个接口，立刻在运行实例上实测**，不要攒一堆猜测最后一起验。
- **动作名/参数从源码提取，不要猜**（正则扫源码字符串）。
- 控制类操作多为**异步**（saga/thunk/事件回环），派发后**等一会儿再读状态**确认。

---

## 第 4 步：工程化为 MCP 插件

1. **摸清宿主（你的助手/Agent）既有约定**：MCP server 配置格式、工具命名、插件管理器、
   设置页接入点、测试约定。**复用，别另起炉灶。**
2. **写自包含的 MCP server**：
   - 一段幂等的 `BOOTSTRAP`，每次命令前重建“三把钥匙”（应对页面刷新）。
   - **按特征定位**接口（扫描签名），不要写死模块 id/混淆名 → 抗版本更新。
   - 工具分层：① 高层语义工具（play/search/…）② **万能逃生口**（调任意 API / 派发任意
     动作 / 执行任意 JS）——让“没单独封装的功能”当场也能做，并为未来扩展留口。
   - 结果结构化：`{ok, action, observedState}`，失败给清晰 `errorCode/message/nextActions`。
3. **接入设置页“安装”流程**：校验目标 → 写配置（注入式才需要装东西；CDP 只需配端口启动）。
4. **三重验证**：单元测试 + 类型检查 + 端到端（真的拉起应用，走 MCP 协议调工具看效果）。

---

## 第 5 步：文档与可维护性

- 写**专项逆向档**（像 `RESEARCH.md`）：原理、控制地图（动作/参数/模块/API 端点）、
  **“更新后自愈指南”**（应用大版本变了如何重新定位）。
- 写 **README**：前提、启动方式、配置、工具清单、排错。
- 标注**风险与边界**：使用条款/封号风险、稳定性、安全（调试端口是本机回环但同机可连）。

---

## 可直接套用的清单（Checklist）

```
[ ] 判断应用类型（CEF/Electron/原生/Web）
[ ] 走控制层决策树，选最靠前可用层；能不改文件就不改
[ ] 做一票否决实验，确认地基成立
[ ] 拿“三把钥匙”：模块入口 / 状态容器 / 网络层（按特征定位）
[ ] 动作名+参数从源码提取；每步在运行实例实测；注意异步时序
[ ] 摸清宿主既有架构并复用；写自包含 MCP server（含万能逃生口）
[ ] 接入安装流程；单测+类型检查+端到端三重验证
[ ] 写专项逆向档（含自愈指南）+ README + 风险边界
```
